# REQ-019: JSON 백업 — 내보내기 / 가져오기

## Status
DONE

## Source
docs/requirements.md §3.9 (저장 방식 — 백업/복원), §7.4 (데이터 영속성)

## Summary
설정 메뉴에서 일기·대화·설정 전체를 단일 JSON 파일로 다운로드(내보내기) / 같은 형식 파일 업로드 후 머지 또는 덮어쓰기(가져오기). PRD가 "강력 권장"한 안전망.

## User Goal
브라우저 캐시 삭제·기기 변경 등 어떤 이유로든 일기 데이터를 잃지 않는다. 사용자가 자기 데이터를 손에 쥘 수 있다.

## Acceptance Criteria
- 내보내기: 클릭 시 `ddalkkak-backup-YYYYMMDD-HHmm.json` 다운로드. 내용은 `{ version: 1, diaries: [...], conversations: [...], settings: {...} }`.
- 가져오기: 파일 선택 → 사전 검증(스키마 + version) → 모드 선택(덮어쓰기 / 머지) → 확인 모달 → 적용.
- 머지 시 같은 `date` 일기 충돌은 사용자에게 선택(현재 유지 / 새로 덮어쓰기) 또는 단순 규칙 1개로 처리 + 결과 토스트.
- 가져오기 후 즉시 화면 갱신.
- 잘못된 형식 파일 업로드 시 에러 안내, 기존 데이터 손상 없음.

## Non-Goals
- 자동 백업 스케줄링 — v2.
- 클라우드 백업(Drive 등) — v2 / Option C.
- 부분 내보내기(특정 달만) — v2.

## Dependencies
REQ-002

## Suggested Scope
- 백업 유틸 모듈 1개 + 설정 화면(또는 기존 더보기 메뉴)에 진입점 추가.

## Expected Affected Areas
- Frontend: 백업 유틸 + 설정 진입점
- Backend: 없음 (Option C에서도 클라이언트만으로 충분)
- Database: 없음
- Tests: 라운드 트립(export→import 시 동일), 잘못된 파일 거부, 머지 vs 덮어쓰기 모드 동작
- E2E: 선택 — export → 가짜 파일로 import → 데이터 동일

## Risk Level
Medium — 잘못된 import가 기존 데이터를 날리면 치명적. 사전 검증·확인 모달 필수.

## Suggested Test Coverage
- export → import(덮어쓰기) → 동일한 storage 상태.
- 손상된 JSON 업로드 → 에러 + 기존 데이터 unchanged.
- version 불일치 시 안전 거부.

## Notes
PRD §7.4 "데이터 손실은 일기 앱에서 가장 치명적". P1이지만 출시 직후 활성화 권장.
