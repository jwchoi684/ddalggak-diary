# REQ-001: 빌드 티어 결정 및 프로젝트 스캐폴드

## Status
DONE

선택 티어: **Option C (Next.js 15 + React 19 + Tailwind 4 + TypeScript)**. Supabase 클라이언트 SDK는 MVP 저장이 localStorage이므로 미설치 — 클라우드 동기화 도입 시점에 별도 REQ.

검증: `npm run build` / `typecheck` / `lint` 모두 PASS. 게이트 리포트는 `.agent-state/architecture-report.md`, `test-report.md`, `review-report.md`, `security-report.md`, `e2e-report.md` 참조.

## Source
docs/requirements.md §10 (기술 스택 제안), §10.5 (권장 빌드 순서), §1.6.7 (레이아웃 원칙)

## Summary
PRD가 명시한 3개 티어(Option A 단일 HTML / Option B React+Vite+Tailwind / Option C Next.js+Supabase) 중 어떤 것을 빌드할지 사용자와 합의하고, 선택된 티어의 최소 스캐폴드를 만든다. 컨테이너 `max-width: 420px` 가운데 정렬 셸까지만 포함하고 실제 기능은 후속 REQ에서 채운다.

## User Goal
이후 모든 기능 REQ가 같은 토대 위에서 구현될 수 있도록 빌드 환경·디렉토리·라우팅 자리·디자인 토큰 자리를 한 번에 정한다.

## Acceptance Criteria
- 사용자가 Option A/B/C 중 하나를 명시적으로 선택했음.
- 선택된 티어로 프로젝트 부팅이 되고 빈 캘린더 자리(placeholder)가 모바일 폭으로 가운데 정렬되어 렌더링됨.
- 디자인 토큰(컬러 §1.6.2, 타이포 §1.6.3)이 한 곳에서 정의되어 있음.
- Option A 선택 시: 단일 HTML 안에 `// ===== Sections =====` 주석 골격이 잡혀 있음 (CLAUDE.md File size 규칙 준수).
- Option B/C 선택 시: `src/design-system/`(또는 동급) 폴더가 생성되어 있음 (CLAUDE.md UI 컴포넌트 재사용 규칙 준수).

## Non-Goals
- 실제 캘린더 / 에디터 / 통계 등 기능 구현 (각 후속 REQ).
- AI 호출 프록시 인프라 구축 (Option B/C에서 REQ-017 시점에 결정).

## Dependencies
없음

## Suggested Scope
- 사용자에게 티어 확인 질문 → 답변 후 스캐폴드 생성.
- 색·폰트 토큰만 정의. 컴포넌트는 REQ-005에서.

## Expected Affected Areas
- Frontend: 빌드 셋업, 진입 HTML/엔트리, 디자인 토큰 정의 파일
- Backend: 없음 (Option C 선택 시 Supabase 프로젝트 생성은 REQ-019 또는 별도)
- Database: 없음
- Tests: 빌드 부트스트랩 smoke test (선택)
- E2E: 빈 캘린더 자리가 렌더된다는 1개 시나리오 (선택)

## Risk Level
High — 티어 선택이 이후 모든 REQ의 구현 방식을 결정. 잘못 고르면 되돌리는 비용이 큼.

## Suggested Test Coverage
- 빌드 명령이 성공한다.
- 모바일·데스크탑 두 폭에서 컨테이너가 420px로 가운데 정렬된다.
